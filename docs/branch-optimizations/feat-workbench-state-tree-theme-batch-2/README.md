## 1. 删除项目 `...` 菜单及对应功能

**改了什么**：删除项目标题栏中的 `...` 按钮，以及由它打开的整理侧边栏、分组、排序和批量归档菜单。

**怎么改的**：移除菜单 Portal、定位计算、菜单状态、排序/分组状态、批量归档回调和中英文文案。项目标题改为静态标签，列表保持后端返回顺序。

**改后怎么样**：项目区域只保留项目切换、新建项目和每行独立归档。单项目归档继续调用真实后端接口，不受本次删除影响。

**验证结果**：桌面 DOM 中不再存在更多菜单按钮或菜单项；新建和单项目归档入口仍存在。

## 2. 消息索引悬浮内容被遮挡

**改了什么**：横条悬浮提示从索引轨道内部移到页面级浮层，并增加对话摘要。

**怎么改的**：鼠标悬浮或键盘聚焦时读取横条位置，通过 React Portal 把固定定位浮层挂到 `document.body`。浮层显示编号、消息标题和最多三行真实内容；Agent 消息优先显示主要 Artifact 摘要，失败消息显示首条错误。

**改后怎么样**：提示不再被 48px 索引滚动容器裁切，也不会被右侧输出框盖住；点击横条仍平滑跳转到对应消息。

**验证结果**：1440px 视口下浮层宽 320px、`z-index: 90`，位于视口内并显示完整问题摘要。

## 3. 完整状态树逻辑与视觉重构

**改了什么**：完整状态树改成“阶段主干 -> Artifact -> Agent 摘要”的六段纵向流程，并修复回退重跑后混入上一轮结果的问题。

**怎么改的**：

- 每个阶段使用独立阶段带，左侧阶段节点、中间本阶段写入对象、右侧 Agent 返回摘要；字段名改为可读名称。
- 节点标注产出来源轮次 `R1/R2/...`，右侧检查器显示阶段状态、来源轮次、执行单元和完整 JSON。
- 前端提交反馈后立即清空目标阶段及下游 `StageRun` 的 input/output/review，并保留目标之前仍有效的上游结果。
- 后端 `record_feedback` 使用字段契约清空目标及下游 task_context 产物，删除对应旧审核，将目标状态设为 `retrying`、下游设为 `queued`。
- 阶段详情接口对 `queued/retrying/rollback/running` 不再返回旧 `latest.output.json`；有效输出按其 metadata iteration 读取匹配的阶段输入。

**改后怎么样**：状态树只显示当前仍有效的科研链路。回退到已完成 Agent 时，目标和下游旧分支立即消失；重新产出的节点按新轮次逐步恢复，上游未失效结果保留原来源轮次。

**验证结果**：新增后端测试覆盖从假设生成回退时的字段、审核和 manifest 状态失效；1440px 浏览器检查 6 个阶段带、19 个当前节点、零节点重叠、零横向滚动。

## 4. 左下角迭代轮次语义

**原有计算**：新任务 `task_context.iteration=1`；每次调用 `record_feedback` 提交修改意见或回退重跑时加 1；普通 Agent 依次执行、人工批准继续和页面刷新都不会增加；分母来自后端 `MAX_WORKFLOW_ITERATIONS`，默认 10。

**是否合理**：该值表示“反馈驱动的科研迭代批次”，不是 Agent 调用次数，因此计算方式合理。原来的 `R1/10` 没有解释 R 的含义，容易被误解为当前阶段序号。

**怎么改的**：标签改为“迭代轮次”，值改为 `1/10` 形式；悬浮说明明确初始任务为第 1 轮、每次反馈回退增加一轮。系统页同步使用相同格式。

**改后怎么样**：用户能区分阶段进度、Agent 数量和科研迭代次数，轮次上限仍由后端统一控制。

## 5. 黑夜与白天主题

**改了什么**：左侧导航增加主题切换按钮，工作台全部主要界面支持亮色和暗色。

**怎么改的**：首次访问跟随系统 `prefers-color-scheme`；用户选择写入 `localStorage`。根节点使用 `data-theme` 和 `color-scheme`，CSS 变量分别定义页面、侧栏、对话、输入控件、状态标签、弹窗、状态树和系统页颜色。

**改后怎么样**：切换无需刷新，按钮图标和文案显示当前主题：白天模式显示太阳，黑夜模式显示月亮；按钮的无障碍名称和悬浮提示说明点击后将切换到哪个主题。刷新页面后保持用户选择。暗色并非简单反色，Artifact、状态和警告仍保留可辨识的语义色。

**验证结果**：浏览器确认亮色为 `lucide-sun`、暗色为 `lucide-moon`，对应的 `data-theme`、浏览器 color scheme、核心表面颜色和刷新持久化均正确；控制台无错误。

## 6. 本轮追加交互修复

### 6.1 精简新建项目入口

**改了什么**：删除左侧任务状态卡最下面的“新建项目”长按钮，只保留项目栏右上角的图标按钮。

**怎么改的**：两个入口原本调用同一个 `createNewProject`，本轮移除重复 JSX，不修改项目创建逻辑。右上角按钮增加“新建项目”自定义悬浮提示、原生 `title` 和无障碍名称。

**改后怎么样**：项目区只有一个稳定的新建入口，任务状态卡只负责展示当前阶段、进度、迭代轮次和最近事件。

**验证结果**：浏览器确认 `.task-meter` 内新建按钮数量为 0，整个项目区“创建新项目”按钮数量为 1。

### 6.2 修复完整状态树滚轮失效

**改了什么**：修复鼠标位于状态树左侧画布、连线或节点上时，滚轮不能向下浏览的问题。

**怎么改的**：React Flow 关闭 `preventScrolling`，不再拦截本组件已经禁用的缩放滚轮事件；`.tree-canvas-scroll` 继续作为唯一双向滚动容器，并增加滚动边界约束和稳定滚动条空间。

**改后怎么样**：状态树仍不可拖拽和滚轮缩放，但鼠标位于空白画布、Artifact 节点或其他左侧内容上时，滚轮都会正常驱动外层状态树向下移动。

**验证结果**：浏览器实测画布区域滚动 518px；随后把鼠标放在“知识空白”Artifact 节点上继续滚动约 401px，容器 `scrollTop` 从 518 增加到约 919。

### 6.3 修正主题按钮表达逻辑

**改了什么**：主题按钮从显示“下一种主题”改为显示“当前主题”。

**怎么改的**：亮色状态渲染太阳和“白天模式”，暗色状态渲染月亮和“黑夜模式”；`aria-label` 与 `title` 仍使用动作语义，如“切换到黑夜模式”。

**改后怎么样**：用户可以直接从图标和按钮文字判断当前界面状态，同时仍能知道点击后的结果。

## 7. 前端按钮功能审计

本轮逐项检查了可见按钮、React 事件处理、前端 API 调用及后端字段消费。当前没有发现“点击后完全没有事件”的普通按钮；状态树节点虽然自身组件没有单独的 `onClick`，但由 React Flow 的 `onNodeClick` 统一处理，不属于空按钮。以下控件有部分实现或语义不准确，建议在后续批次处理：

| 控件 | 审计结论 | 当前真实行为 | 问题与建议 |
| --- | --- | --- | --- |
| Agent 消息中的阶段说明横条 | 交互反馈弱 | 点击只切换左侧当前阶段和主输入框的反馈目标。 | 外观看起来像“打开详情”，但不会打开状态树或跳到详情。建议改成普通说明文字，或明确改为点击打开对应状态树节点。 |
| 系统页“刷新” | 部分实现 | 只重新请求 Artifact、版本列表和最近版本差异。 | 页面同时展示的 context、阶段状态、事件和附件没有重新请求。建议实现整任务刷新，或改名为“刷新产物与版本”。 |
| “输出深度”三个分段按钮 | 部分生效 | 写入 `output_detail_level`，目前后端主要把它映射成研究计划的 `plan_depth`。 | 不能稳定改变前面所有 Agent 的输出长度。建议让五个 Agent 统一消费该字段，或改名为“计划深度”。 |
| “记忆”低/中/高 | 功能存在，说明不准确 | 后端只按档位控制重跑时注入多少条历史反馈和同阶段审核问题。 | 不会改变版本、证据或 Artifact 的持久化保留量。建议把文案改成“反馈上下文范围”，不要描述为保留更多版本和证据。 |
| “替我审批” | 功能存在，说明不准确 | `hybrid` 模式只在预先标记的证据梳理、研究计划、总控最终输出阶段请求人工确认。 | 不是“检测到风险操作后请求批准”。建议直接说明“仅在关键阶段询问”。 |
| “完全自动” | 功能存在，说明不准确 | 合格结果由 Review Gate 规则自动接受，低质量结果仍进入重试或人工选择流程。 | 不是“完全由模型自行审批”。建议改成“由总控规则自动审批合格输出”。 |
| 推理强度 | 功能生效但受环境限制 | 始终影响模型最大输出 token；只有后端开启 `QWEN_ENABLE_THINKING=true` 时，高/超高才启用模型 thinking。 | 建议根据健康检查中的 `thinking_enabled` 显示是否真实开启深度思考，避免把 token 档位误认为一定启用 thinking。 |

已经确认具有真实行为的按钮包括：项目新建/切换/单项目归档、文件上传、问题发送与反馈重跑、人工批准、查看 JSON、消息索引跳转、状态树节点与退出、系统导出、页面导航、中英文切换和明暗主题切换。

## 8. 品牌图标与浏览器标签页

**改了什么**：使用负责人提供的六相拓扑环图标替换左侧品牌栏和对话区 AI 输出原有机器人图标，同时替换浏览器标签页图标并补齐网页安装图标。

**怎么改的**：

- 负责人提供的 1254×1254 黑色版和 1024×1024 白色版分别归档为 `docs/brand/eurekaloop-logo-black-master.png`、`docs/brand/eurekaloop-logo-white-master.png`，不进行 AI 重绘；母版不放在 `public`，避免被 Vite 复制进生产包。
- 黑色母版中的棋盘格实际烘焙在 RGB 中，因此先按中性高亮阈值、柔和边缘和去白边规则提取真实 Alpha；白色母版已有 Alpha，清理极低透明度残留后直接使用其原始白色主体。
- 两个版本分别按有效主体边界统一为相同方形留白和 512px 构图，再以高质量双三次插值生成 192、180、64、32 和 16 像素资源。
- 普通黑白图标四角 Alpha 均为 0，中心孔洞和外围背景同样透明；白天模式直接使用负责人提供的黑色版，黑夜模式直接使用负责人提供的白色版。
- maskable 图标按 PWA 规范保留深色不透明承载面和更大安全边距。这是系统裁切图标的专用例外，不用于普通网页和品牌栏显示。
- 左侧品牌栏按当前主题切换 64px 透明版本；`index.html` 显式声明亮暗两套 32px、16px favicon 和 180px Apple Touch Icon；manifest 注册透明 192px、512px 和 maskable 512px 图标。
- Agent 与总控消息统一使用 34px 透明品牌头像，并按当前主题切换黑色或白色版本；用户消息继续显示“你”，保持角色方向和身份辨识。
- 删除旧的紫色闪电 `favicon.svg`，避免部署后浏览器继续命中旧品牌资源。

**改后怎么样**：工作台左上角、AI/总控输出头像和浏览器标签页不再出现旧机器人或底色方块；亮色界面显示新黑色图标，暗色界面显示新白色图标。添加到桌面和 PWA 安装入口继续使用符合系统裁切规则的专用图标。

**验证结果**：黑白两套 512/192/180/64/32/16 图标四角 Alpha 均为 0；浏览器确认品牌容器和 AI 消息头像背景均为 `rgba(0, 0, 0, 0)`，暗色主题只显示白色透明版本、亮色主题只显示黑色透明版本。当前恢复项目中的 7 条 AI/总控消息全部显示新头像，3 条用户消息仍显示“你”。页面声明亮暗两套 favicon、180px Apple Touch Icon 和 manifest，控制台无资源加载错误。生产构建已包含新图标声明。

## 人工验收顺序

1. 启动前后端并进入已有项目，确认项目标题旁不再有 `...`，项目切换、新建项目和单项目归档仍可用。
2. 将鼠标放到工作区左侧的消息索引横条，确认浮层展示对应对话摘要，且不会被中间对话区域遮挡；点击后应跳转到对应消息。
3. 打开完整状态树，确认六个阶段全部展开，并能沿“阶段 -> 写入对象 -> 返回摘要”阅读；点击任意节点后，右侧检查器内容应同步变化。
4. 对已完成 Agent 提交改进意见并回退重跑，确认目标阶段和下游旧节点立即消失，重新执行后只展示新一轮有效结果。
5. 查看左下角“迭代轮次”，新任务应从 `1/10` 开始；只有提交反馈回退时增加，普通批准继续不增加。
6. 切换黑夜/白天主题并刷新页面，确认工作台、输入框、弹窗、状态树和系统页均切换且主题选择被保留。
7. 确认任务状态卡底部不再出现新建项目按钮；项目栏右上角新建图标悬浮后显示“新建项目”。
8. 打开已有完整结果的状态树，把鼠标分别放在空白画布和节点上滚动，确认两处都能纵向浏览六个阶段。
