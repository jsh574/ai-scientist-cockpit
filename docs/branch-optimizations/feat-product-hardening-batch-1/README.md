# feat/product-hardening-batch-1 优化清单

## 分支信息

- 基线：`origin/main` 的 `c84b039 allow operator choice at quality gates`
- 当前分支：`feat/product-hardening-batch-1`
- 批次目标：修复任务输入语义、持久化、附件、状态树、真实健康状态、运行反馈和模型调用稳定性。
- 提交状态：尚未提交、尚未推送、尚未合并，等待负责人验收。

## 1. 主输入框重复创建任务

**改了什么**：区分“新科学问题”和“已有项目修改意见”。

**怎么改的**：首次提交调用 `POST /api/tasks`；已有 `task_id` 后显示反馈目标模块，发送内容改走 `POST /api/tasks/{id}/feedback`，再从目标阶段继续执行。人工审核仍保留在对应模块消息尾部。

**改后怎么样**：同一项目内的后续输入保持原 `task_id` 并增加 iteration，不会从问题理解阶段无条件新建任务。新任务只能通过“创建新项目/新建项目”进入。

**验证结果**：后端测试确认反馈后 `task_id` 不变、iteration 增加；浏览器确认恢复后的项目显示“反馈目标”和“输入修改意见”。

## 2. 项目刷新丢失与归档无效

**改了什么**：项目列表由后端恢复，批量归档和单项目归档都真实写入 manifest。

**怎么改的**：新增任务列表与归档 API；页面启动后读取 context、events、attachments 和六阶段详情。使用 `Promise.allSettled` 隔离旧格式任务，并修复 React 严格模式下首次恢复结果被丢弃的问题。

**改后怎么样**：刷新后可恢复未归档项目；每个项目行可单独归档，项目菜单仍可批量归档；归档当前项目后自动切换到剩余项目，无剩余项目时创建新草稿；单个异常任务不会拖垮其他项目。

**验证结果**：自动测试覆盖归档过滤/恢复；浏览器确认项目行存在独立归档入口。未使用用户现有项目执行破坏性归档测试。

## 3. 附件按钮只有文件名

**改了什么**：附件改为真实上传和任务级持久化。

**怎么改的**：前端保存 `File[]`，校验类型和大小；后端限制 UTF-8 的 `.txt/.md/.csv/.json`，写入 `attachments/` 和索引，并把文本摘要注入 `user_input.question_description`。共享模型网关把该背景加入每次真实模型请求的用户消息，若 Prompt 已含相同内容则去重。

**改后怎么样**：Agent 可读取附件背景；刷新后可恢复附件列表；首次上传失败也不会重复创建任务，文件可在下一次反馈时重试。

**验证结果**：自动测试覆盖文件落盘、上下文注入、不支持格式和非法编码；模型网关测试用唯一标记 `ATTACHMENT_SENTINEL_7F31` 证明附件文本真实进入 OpenAI 兼容请求的用户消息。

## 4. 推理、审批和记忆只改变前端

**改了什么**：三类控制项接入任务创建和反馈迭代。

**怎么改的**：反馈请求携带 `mode`、`reasoning_level`、`memory_level`；后端在重跑前写回上下文。推理等级影响模型预算，记忆等级决定纳入的反馈和审核历史数量。thinking 只有在 `QWEN_ENABLE_THINKING=true` 且等级为 high/ultra 时开启，默认关闭。

**改后怎么样**：切换设置后，下一轮 Agent 使用新的运行参数；恢复项目时控件同步后端状态。

**验证结果**：自动测试覆盖设置写回和低/中/高记忆历史范围。

## 5. Agent/API 状态具有误导性

**改了什么**：系统页不再使用静态 `ready` 或仅按源码路径计数。

**怎么改的**：健康接口返回每个 Agent 的源码可用性、凭据要求、凭据状态和真实 ready；API 列表按 capabilities 渲染 ready/offline/unknown。

**改后怎么样**：缺少模型密钥时显示降级和具体原因；规则引擎可独立显示就绪；系统页列出 5 个 Agent 的执行模式。

**验证结果**：自动测试覆盖无凭据时模型 Agent 不就绪、规则引擎仍可用。

## 6. 消息索引位置与状态树可读性

**改了什么**：消息索引移入工作区域；PC 完整状态树改为六阶段全量展开。

**怎么改的**：索引位于项目侧栏右侧，仅有消息时显示。状态树同时渲染六阶段主干、全部 Artifact 和已有 Agent 返回摘要；stage、artifact、detail 使用稳定高度和清晰字号，画布按纵向拓扑滚动，不再用 fitView 把全部内容压小。右侧检查器显示完整输入、输出、审核和 JSON。

**改后怎么样**：项目管理与对话跳转层级明确；打开弹窗即可查看完整流程，不需要逐阶段点击展开；分支不再互相重叠，`question_card` 等标识不会被下方摘要遮挡，完整内容仍可在检查器中读取。

**验证结果**：1280px PC 视口下画布 `clientWidth=867`、`scrollWidth=867`，无横向滚动；六阶段、Artifact、Agent 摘要和检查器可见，退出按钮正常。

## 7. 移动端工作区被侧栏挤出首屏

**改了什么**：窄屏侧栏压缩为顶部项目栏，主工作区保持单屏结构。

**怎么改的**：隐藏移动端状态树摘要和任务仪表，项目横向滚动，导航改为四列；科研问题卡片与约束改为单列；对话区内部滚动，输入框固定在工作区底部。

**改后怎么样**：390x844 视口下管理区约 175px，页面总高度等于视口，代表性问题和输入框同时可访问，无页面级横向溢出。

**验证结果**：浏览器实测 `scrollHeight=844`、主工作区宽度 390、初始对话 `scrollTop=0`。

## 8. 模型名称误导与依赖缺口

**改了什么**：移除写死且只会关闭菜单的 `GPT-5.5`；补充上传接口依赖。

**怎么改的**：`GET /api/health` 返回当前模型名，推理控件和系统状态从健康接口动态读取；推理菜单只保留实际生效的强度选项；`backend/requirements.txt` 增加 `python-multipart`。

**改后怎么样**：当前配置为 qwen3.7-max 时，界面显示 `qwen3.7-max · 高`，以后后端切换模型时前端无需改代码；全新环境可正常注册 FastAPI 多文件上传路由。

**验证结果**：后端启动成功，附件路由可用，前端生产构建成功。

## 9. 模型运行时缺少进度感知

**改了什么**：每条正在生成的模型消息显示累计运行时间，完成后保留实际用时。

**怎么改的**：消息记录开始时间和 `durationMs`，前端每秒刷新运行标签，并按秒、分秒、小时分秒格式化；超过 120 秒使用警示色。

**改后怎么样**：用户可以区分“仍在运行”和“页面无响应”。历史响应也能显示后端记录的真实耗时，例如知识整合 `用时 15分40秒`。

**验证结果**：浏览器检查运行标签和历史完成用时；后端 `metadata.duration_ms` 可恢复到消息记录。

## 10. 对话头像方向不统一

**改了什么**：模型输出统一在左侧显示 Bot 头像，用户输入统一在右侧显示用户头像。

**怎么改的**：总控和五个 Agent 共用模型消息布局；用户消息使用独立的右对齐 flex 顺序，消息框随发送方对齐。

**改后怎么样**：阅读方向与主流对话产品一致，连续的模型输出不会在左右两边跳动。

**验证结果**：浏览器确认 Agent/总控头像在左侧，用户消息样式按右侧布局渲染。

## 11. 个别阶段极慢或超时

**改了什么**：收紧模型调用的默认重试和 thinking 策略，并暴露运行参数用于诊断。

**怎么改的**：SDK 默认 `LLM_MAX_RETRIES=0`，知识整合默认 `KNOWLEDGE_LLM_MAX_ATTEMPTS=1`；high token 上限调整为 6144，ultra 才使用完整预算；high/ultra 不再自动开启 thinking。健康接口增加 timeout、retry、thinking 和知识整合尝试次数。

**改后怎么样**：单次 120 秒超时不会再因 SDK 默认重试直接翻倍，知识整合也不会默认重复整套模型步骤。系统页可直接看到实际超时与重试设置。

**验证结果**：历史记录确认慢任务曾由多模型步骤和重试累积到 15分40秒，并出现 `APITimeoutError`；新增测试确认 high 默认不启用 thinking、SDK 重试为 0。该问题不是前后端接口断开。
