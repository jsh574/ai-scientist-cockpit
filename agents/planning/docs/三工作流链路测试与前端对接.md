# Planning Agent 三工作流链路测试与前端对接

本文说明如何验证已经发布到 Dify 的三个 Workflow，并定义提供给前端开发者的中间结果契约。测试代码不替换现有 `run_planning_agent`，因此不会影响当前可运行的 Workflow C 生产路径。

## 1. 测试目标

测试链路固定为：

```text
模块 5 输入
  -> 本地按 selection_score 构建一个或多个 hypothesis_evidence_package
  -> 每个 hypothesis 独立进入下面的 A -> B -> C 子链
  -> Workflow A 并发生成 3 个 design_candidate
  -> Workflow B 评分、解释并选择
  -> accept: Workflow C 生成最终 plan_result
  -> revise_once: 最多带 revision_instruction 重跑一次 A/B
  -> feedback_required / human_review / failed: 停止，不调用 C
```

默认候选模式：

- `minimum_viable`
- `high_information`
- `resource_efficient`

这段本地代码只负责测试和串接 Planning Agent 内部模块。上游负责提出科学假设，A 不生成新假设，而是为同一 `hypothesis_id` 生成不同实验设计；证据检索和证据映射仍由上游负责。批量测试按本地 selection score 排序，但不改变假设内容。

## 2. 文件说明

- `planning_agent/workflow_api.py`：通用 Dify Workflow API 客户端，保留 End 节点的全部输出。
- `planning_agent/workflow_chain.py`：A -> B -> C 测试编排、一次受控修订、上下文长度控制。
- `planning_agent/workflow_chain_cli.py`：命令行入口。
- `planning_agent/workflow_chain_report.py`：生成独立 HTML 测试报告，不依赖现有产品前端。
- `tests/test_workflow_chain.py`：完全离线的请求、状态路由、修订循环和 HTML 转义测试。
- `.env.example`：三个 Dify App 的配置模板，不包含真实密钥。

## 3. 配置 Dify API Key

Dify 的 API Key 归属于具体 App。即使三个 Workflow 部署在同一个 Dify 实例，也必须分别在每个 Workflow 的“访问 API”页面创建 Key。

复制 `.env.example` 中需要的变量到本地 `.env`，不要把 `.env` 提交到 Git：

```env
DIFY_API_URL=https://your-dify-host.example.com
DIFY_WORKFLOW_A_API_KEY=app-key-for-candidate-generator
DIFY_WORKFLOW_B_API_KEY=app-key-for-judge-selector
DIFY_API_KEY=app-key-for-research-planning-workflow-c
DIFY_CHAIN_RESPONSE_MODE=streaming
DIFY_CHAIN_TIMEOUT_SECONDS=300
```

兼容规则：

- A 使用 `DIFY_WORKFLOW_A_API_URL`，为空时回退到 `DIFY_API_URL`。
- B 使用 `DIFY_WORKFLOW_B_API_URL`，为空时回退到 `DIFY_API_URL`。
- C 使用 `DIFY_WORKFLOW_C_API_URL`，为空时回退到 `DIFY_API_URL`。
- C Key 优先使用 `DIFY_WORKFLOW_C_API_KEY`，为空时回退到现有的 `DIFY_API_KEY`。
- A/B 不回退到 `DIFY_API_KEY`，因为不同 Dify App 不能共用一个 App Key。

先确认配置，不发送请求，也不会打印 Key：

```powershell
.\.venv\Scripts\python -m planning_agent.workflow_chain_cli --print-targets
```

三个对象的 `configured` 都应为 `true`，endpoint 应为对应 Dify 实例的 `/v1/workflows/run`。

## 4. 运行真实 smoke test

不带批量参数时，命令只测试指定 hypothesis，或默认测试本地评分最高的一个。单个假设在无修订情况下产生 5 次 Dify 请求：A 三次、B 一次、C 一次。

```powershell
.\.venv\Scripts\python -m planning_agent.workflow_chain_cli --sample
```

让输入中的全部假设分别完整经过 A/B/C，并允许 3 个假设并发：

```powershell
.\.venv\Scripts\python -m planning_agent.workflow_chain_cli `
  --input samples\input\module5_input_rag_batch.json `
  --all-hypotheses `
  --max-parallel-hypotheses 3 `
  --output samples\test-artifacts\rag-batch-abc.json `
  --html samples\test-artifacts\rag-batch-abc.html
```

该 RAG 样例包含 3 个假设。无修订且全部 accept 时，总请求数为 15：每个假设 3 次 A、1 次 B、1 次 C。外层并发 3 与 A 内层三变体并发叠加，峰值可能达到 9 个 A 请求；如果 Dify 并发能力有限，将 `--max-parallel-hypotheses` 改为 1 或 2。

指定真实模块 5 输入和 hypothesis：

```powershell
.\.venv\Scripts\python -m planning_agent.workflow_chain_cli `
  --input samples\input\module5_input_sample.json `
  --hypothesis-id hyp_001
```

指定输出位置：

```powershell
.\.venv\Scripts\python -m planning_agent.workflow_chain_cli `
  --sample `
  --output samples\test-artifacts\chain-smoke.json `
  --html samples\test-artifacts\chain-smoke.html
```

默认在 `samples/test-artifacts/` 生成同名 `.json` 和 `.html`。该目录已被 `.gitignore` 忽略。HTML 可以直接用浏览器打开，按 A 候选、B 选择和 C 最终计划展示中间 JSON、run ID、耗时及错误。

退出码：

| 退出码 | report.status | 含义 |
|---:|---|---|
| 0 | `success` | B 已接受候选且 C 返回有效 `plan_result`。 |
| 2 | `requires_action` | 需要人工评审、上游补充证据或修订次数已用完；C 未执行。 |
| 1 | `partial_success` / `failed` | 批量中部分假设失败，或配置、HTTP、输出 JSON、上下文校验失败。 |

## 5. 先运行离线测试

离线测试不会调用 Dify，也不会消耗模型 token：

```powershell
.\.venv\Scripts\python -m pytest
```

只运行本次新增测试：

```powershell
.\.venv\Scripts\python -m pytest tests\test_workflow_chain.py -q
```

覆盖的关键路径：

- Dify blocking 响应保留多个 End 输出。
- A 的三个候选都进入 B。
- B `accept` 后，`selected_design` 和 `design_selection` 进入 C 的 `planning_constraints`。
- B `revise_once` 最多重跑一次 A/B。
- B `human_review` 时绝不调用 C。
- HTML 对模型输出做转义，避免把 JSON 内容当作脚本执行。

## 6. 中间结果 JSON 契约

测试报告根结构：

```json
{
  "schema_version": "planning_workflow_chain_test_v1",
  "task_id": "task_short_001",
  "iteration": 1,
  "hypothesis_id": "hyp_short_001",
  "status": "success",
  "decision": "accept",
  "next_action": "continue_to_product",
  "started_at": "2026-07-22T00:00:00+00:00",
  "completed_at": "2026-07-22T00:01:00+00:00",
  "duration_seconds": 60.0,
  "stages": [],
  "intermediate_results": {
    "candidate_rounds": [],
    "selection_rounds": []
  },
  "final_result": {},
  "errors": []
}
```


批量模式使用独立根 schema，并在 `hypothesis_runs[]` 中保留每个假设的上述完整报告：

```json
{
  "schema_version": "planning_workflow_chain_batch_test_v1",
  "task_id": "task_rag_batch_001",
  "status": "success",
  "next_action": "continue_to_product",
  "configuration": {
    "max_parallel_hypotheses": 3,
    "selection_order": "selection_score_desc"
  },
  "hypothesis_ids": ["hyp_rag_001", "hyp_rag_002", "hyp_rag_003"],
  "summary": {
    "total": 3,
    "success": 3,
    "requires_action": 0,
    "failed": 0
  },
  "hypothesis_runs": [
    {"schema_version": "planning_workflow_chain_test_v1", "hypothesis_id": "hyp_rag_001"}
  ],
  "errors": []
}
```
前端建议只依赖以下稳定字段：

| 展示区域 | 字段 |
|---|---|
| 总状态 | `status`, `decision`, `next_action`, `duration_seconds` |
| 阶段时间线 | `stages[].stage_id`, `status`, `workflow_run_id`, `elapsed_time` |
| A 候选对比 | `intermediate_results.candidate_rounds[].candidates[]` |
| A 护栏 | `intermediate_results.candidate_rounds[].guardrail_reports[]` |
| B 评分解释 | `intermediate_results.selection_rounds[].design_selection` |
| B 确定性校验 | `selection_guardrail_report` |
| C 最终计划 | `final_result` |
| 阻断原因 | `errors[]`, `next_action` |

`stages[].outputs` 保留 Dify End 原始语义并已将 JSON 字符串解码为对象，主要用于调试。产品前端不要展示模型隐藏思考，也不要依赖 `text_chunk` 内容。

## 7. 状态路由

| B decision | 本地行为 | 前端行为 |
|---|---|---|
| `accept` | 校验 `selected_design` 后调用 C。 | 继续展示最终计划。 |
| `revise_once` | 将 `revision_instruction` 放入 `planning_constraints.candidate_revision`，最多重跑一次 A/B。 | 展示第二轮候选；若仍未 accept，显示需要处理。 |
| `feedback_required` | 停止，不调用 C。 | 显示 `feedback_tasks`，交给上游或用户补充。 |
| `human_review` | 停止，不调用 C。 | 显示候选评分与冲突，进入人工选择。 |
| `failed` | 停止，不调用 C。 | 显示 guardrail/API 错误和 run ID。 |

## 8. Workflow C 上下文控制

C 的 `planning_constraints` 在当前 DSL 中上限是 12000 字符。测试器先尝试传入完整：

```json
{
  "...original_constraints": "...",
  "selected_design": {},
  "design_selection": {}
}
```

如果超过上限，只压缩 B 的候选评审明细，保留完整 `selected_design`、decision、selected candidate、meta review 和 limitations，并在 C stage 的 `context_control.strategy` 标记 `compact_selection_context`。压缩后仍超限会失败，而不是静默截断计划设计。

## 9. 真实环境验收清单

首次发布后至少保存以下 5 组报告：

1. 短样例 `accept`：A 三个候选、B 选择、C 最终计划均成功。
2. 一个真实模块 5 输入：核对 evidence/source ID 没有越过 allowlist。
3. `human_review` 或接近候选：验证 C 没有被调用。
4. 故意使用错误 A Key：报告应包含 Workflow A HTTP 错误且不泄露 Key。
5. 故意传不存在的 hypothesis ID：应在本地失败且不产生任何 Dify 请求。

验收时记录 JSON 中的 `workflow_run_id`，可在 Dify 后台按 ID 对照节点日志。不要把真实 Key、Dify 网页调试导出或含敏感研究数据的测试报告提交到 Git。

## 10. Thinking 模型与结构化输出对接规则

### 事故与根因

2026-07-22 的首次真实 A/B/C 测试中，Dify 后台显示 Workflow A 三次 `success`，但 B/C 没有执行。流式节点诊断确认：A 的 LLM `text` 长度为 4521，包含 `<think>...</think>`；同一节点的 `structured_output` 已经是包含全部 20 个字段的正确对象。Guard Code 当时选择 `text` 并直接执行 `json.loads()`，解析失败后回退为 `{}`，所以业务 guard 报告 identity、schema 和必填字段全部缺失。

这属于接口选择 BUG，不是模型未生成结果。Dify `status=success` 只表示节点正常执行结束，不代表 End 输出通过业务校验。

### 强制对接规则

- 开启 `structured_output_enabled` 的 LLM，其下游 Code 节点必须优先选择 `[llm_node_id, structured_output]`。
- thinking-enabled 模型的 `text` 可能包含 `<think>`、Markdown 围栏或其他展示文本，不能直接交给 `json.loads()`。
- Code 入参应体现对象语义，例如 `candidate_payload: dict`、`selection_payload: dict`，不要把结构化对象命名为 `*_text`。
- 只有上游没有结构化输出时才允许读取 `text`；必须先清理 thinking/围栏，并把解析失败作为显式错误，不能静默变成空对象继续执行。
- 本地 hard gate 必须保留。执行层 success 之后仍要校验 schema、hypothesis/candidate identity、evidence allowlist 和 guardrail。

本次修复：

| Workflow | 修复后的 selector | 旧的错误 selector |
|---|---|---|
| A | `design_candidate_llm.structured_output` | `design_candidate_llm.text` |
| B | `judge_selector.structured_output` | `judge_selector.text` |

`tests/test_modular_workflow_assets.py` 使用 YAML 解析器锁定以上契约，避免后续导出或手工编辑时回归。修改本地 YML 不会自动更新 Dify 中已经发布的 App：必须重新导入 A/B/C，或在画布中同步修改对应节点并重新发布，然后再运行 `--sample` smoke test。

### 系统身份字段所有权

2026-07-22 第二次 smoke test 中，A 的两个候选因为模型生成了错误的 `hypothesis_id` 被丢弃；B 又因为模型生成的 `task_id` 与 Start 输入不一致而被 Guard 强制改成 `failed`。B 的 `selection_context` 当时甚至没有包含 `task_id` 和 `iteration`，却要求模型输出这两个字段。

这类字段属于工作流路由元数据，不属于模型推理结果。统一规则如下：

- A 的 `hypothesis_id`、`variant_mode`、`candidate_id` 由 Guard 根据 Start 输入覆盖。
- B 的 `task_id`、`iteration`、`hypothesis_id` 由 Guard 根据 Start 输入覆盖；Prepare Context 同时向模型提供这些值，降低无效生成。
- C 的 `schema_version`、`agent_name`、`task_id`、`iteration`、`hypothesis_id` 由最终 Contract Code 节点覆盖。
- 发生覆盖时不把它当作业务失败，而是在 `normalized_identity_fields` 中记录，便于观测模型稳定性。
- `selected_candidate_id`、`candidate_reviews[].candidate_id`、evidence ID 和 source ID 属于语义引用，不能静默覆盖；未知、缺失或重复引用仍是 hard-gate 失败。

Workflow C 同样必须使用结构化通道：

| Consumer | 正确输入 |
|---|---|
| Generate Final Plan Fast | `normalize_evidence.normalized_evidence_context`，其中包含 B 的 `selected_design`（若有） |
| Final Contract Code | `full_plan.structured_output` |
| End | `final_contract.plan_result`、`final_contract.contract_report` |

快速终稿 LLM 必须关闭 thinking，使用 `response_format=json_object` 和严格 structured output。C 从四次串行 LLM 缩减为一次，但本地 hard gate 和最终 Contract Code 不变。

本地链路也必须区分 Dify 执行状态与 C 的业务状态：只有 `plan_result.status=success` 且 `plan` 为非空对象时，链路总状态才是 `success`；`partial_success` 进入人工复核，`failed` 停止并返回 `inspect_failure`。

### 候选过滤与 B 格式恢复

Workflow A 的执行成功和候选可用必须分开处理：

- 只有 `guardrail_report.passed=true` 且候选自身 `status != failed` 的候选进入 Workflow B。
- 被拒候选仍完整保留在 `stages[].runs`，状态为 `rejected`，不会从调试报告中消失。
- `candidate_generation` 增加 `accepted_candidate_count` 和 `rejected_candidate_count`；只要不是全部候选可用，阶段状态就是 `partial_success`。
- `candidate_rounds[].guardrail_reports` 保留所有 A 调用的 Guard 结果，而不仅是通过的候选。

Workflow B 对模型格式错误提供一次有界恢复：

- 默认最多额外重试一次，由 `DIFY_WORKFLOW_B_MAX_FORMAT_RETRIES=1` 控制；允许范围是 0 到 2。
- 仅当 `decision=failed` 且 `selection_guardrail_report.passed=false` 时重试。
- 重试输入会包含 `planning_constraints.selection_format_retry`，其中记录 attempt、上一轮 issues 和完整根对象提示。
- `human_review`、`feedback_required` 和有效的 `revise_once` 不属于格式错误，不触发该重试。
- 每次 B 调用都保留在 `stages[]` 和 `selection_rounds[]`，使用 `attempt` 区分；重试后仍失败才写入顶层 `errors` 并停止。

B 的 `candidate_reviews.items` 和七维 `scores` 已使用完整 JSON Schema：字段不允许遗漏或扩展，评分范围固定为 0 到 1。Guard 会再次执行同样的运行时校验，防止模型提供商只把 schema 当作提示而不强制执行。

## 11. 合并到产品前的边界

当前测试器刻意不修改 `planning_agent/service.py`。真实 smoke test 通过后，再将同样的 A/B/C 调用逻辑接入 Planning Agent 自己的服务层，并保持对外模块 5 返回的 `metadata + payload + self_review` 契约不变。产品前端可以将本文的报告字段映射为内部进度视图，但不应把测试报告根结构直接替换成 Agent 的正式输出契约。
