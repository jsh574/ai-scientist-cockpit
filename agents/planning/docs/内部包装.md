## 内部证据包派生规则

模块 5 可在正式生成研究计划前，将输入字段派生为内部 `hypothesis_evidence_packages`，但该结构不要求上游直接提供。

```JSON
{
  "hypothesis_evidence_packages": [
    {
      "hypothesis_id": "string",
      "hypothesis": "string",
      "rationale": "string",
      "target_variables": ["string"],
      "expected_observation": "string",
      "validation_idea": "string",
      "scores": {
        "initial_scores": {},
        "evidence_strength_score": 0.75
      },
      "evidence_subset": {
        "supporting_evidence": [],
        "opposing_evidence": [],
        "uncertain_evidence": []
      },
      "source_literature": [],
      "knowledge_gaps": [],
      "limitations": ["string"],
      "needs_more_evidence": true
    }
  ]
}
```

派生逻辑：

```Plaintext
1. 以 hypothesis_cards[].hypothesis_id 为主键。
2. 用 evidence_map[].hypothesis_id 找到该假设的 supporting/opposing/uncertain evidence_ids。
3. 用 evidence_ids 回查 evidence_cards，形成 evidence_subset。
4. 用 evidence_cards[].source_literature_id 回查 literature_cards，形成 source_literature。
5. 用 hypothesis_cards[].related_gap_ids 回查 knowledge_gaps。
6. 结合 initial_scores、evidence_strength_score、needs_more_evidence 对假设进行排序或筛选。
```

排序建议：

```Plaintext
优先选择 testability 高、relevance 高、evidence_strength_score 中高、risk 可控的假设。
若 needs_more_evidence 为 true，不应直接淘汰，而应在研究计划中转化为补充验证任务。
```
