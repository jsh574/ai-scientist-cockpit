# 数据规范 v0\.1

## 1\. 设计目标

本系统采用“总控模块 \+ 多 Agent 模块”的结构。用户输入科学问题后，总控模块维护统一任务上下文，并按流程依次调用 5 个 Agent：

```Plaintext
用户输入
→ 模块 1：问题理解
→ 模块 2：知识整合
→ 模块 3：候选假设生成
→ 模块 4：证据梳理
→ 模块 5：研究计划输出
→ 总控最终审核
```

数据规范：

```Plaintext
每个模块只读取自己需要的数据对象，只写入自己负责的新数据对象。
总控负责维护完整上下文、检查字段是否齐全、判断是否可以进入下一步。
```

## 2\. 核心机制

系统不采用“上一个模块的全部输出直接丢给下一个模块”的方式，而是采用统一的 `task_context`。

总控维护完整任务上下文：

```Plaintext
task_context
├── user_input
├── question_card
├── literature_cards
├── evidence_cards
├── knowledge_gaps
├── hypothesis_cards
├── evidence_map
├── research_plan
├── reviews
└── versions
```

每个模块执行时：

```Plaintext
1. 从 task_context 中读取自己需要的字段
2. 生成本模块负责的数据对象
3. 进行模块自评
4. 将结果交给总控
5. 总控校验通过后写回 task_context
```

## 3\. 全局任务上下文 task\_context

`task_context` 是总控维护的总档案。

```JSON
{
  "task_id": "task_001",
  "mode": "auto",
  "current_stage": "question_understanding",
  "iteration": 1,
  "user_input": {
    "original_question": "阿尔茨海默病的关键致病机制是什么？",
    "user_constraints": {
      "language": "zh",
      "domain_preference": "biomedicine",
      "max_hypotheses": 5
    }
  },
  "question_card": null,
  "literature_cards": [],
  "evidence_cards": [],
  "knowledge_gaps": [],
  "hypothesis_cards": [],
  "evidence_map": [],
  "research_plan": null,
  "reviews": [],
  "versions": []
}
```

字段说明：

|字段|含义|
|---|---|
|`task_id`|当前研究任务 ID|
|`mode`|执行模式，`auto` 表示自动执行，`manual` 表示每步人工审核|
|`current_stage`|当前执行阶段|
|`iteration`|当前迭代轮次|
|`user_input`|用户原始输入和约束|
|`question_card`|问题理解模块输出|
|`literature_cards`|文献卡片列表|
|`evidence_cards`|证据卡片列表|
|`knowledge_gaps`|知识空白列表|
|`hypothesis_cards`|候选假设列表|
|`evidence_map`|假设与证据绑定关系|
|`research_plan`|最终研究计划|
|`reviews`|各阶段审核结果|
|`versions`|多轮迭代版本记录|

## 4\. 统一模块响应格式

所有 Agent 的输出建议统一包一层外壳，方便总控处理。

```JSON
{
  "metadata": {
    "task_id": "task_001",
    "agent_id": "question_understanding_agent",
    "stage": "question_understanding",
    "iteration": 1,
    "status": "success"
  },
  "payload": {},
  "self_review": {
    "passed": true,
    "overall_score": 0.86,
    "threshold": 0.75,
    "dimension_scores": {},
    "issues": [],
    "suggestions": []
  }
}
```

字段说明：

|字段|含义|
|---|---|
|`metadata`|统一元信息，总控主要读取这一部分|
|`payload`|模块实际输出内容，不同模块结构不同|
|`self_review`|模块自评结果|

注意：

```Plaintext
模块自评负责判断“内容质量好不好”。
总控校验负责判断“格式是否合规、字段是否齐全、能不能进入下一步”。
```

## 5\. 模块 1：问题理解 Agent

### 5\.1 目标

将用户输入的原始科学问题，转化为结构化、标准化、可检索、可检验、可迭代的研究问题卡片 `question_card`。

### 5\.2 输入

读取 `task_context.user_input`。

```JSON
{
  "original_question": "阿尔茨海默病的关键致病机制是什么？",
  "user_constraints": {
    "language": "zh",
    "domain_preference": "biomedicine"
    "output_detail_level": "standard"
  }
}
```

|字段|类型|是否必需|说明|
|---|---|---|---|
|`original_question`|string|必需|用户输入的原始科学问题，是问题理解 Agent 需要解析的核心内容。|
|`user_constraints`|object|可选|用户对输出结果的附加约束。|
|`language`|string|可选|输出语言，例如中文 `zh`、英文 `en`。|
|`domain_preference`|string|可选|用户指定的领域偏好，例如 `biomedicine`、`materials_science`、`computer_science`。|
|`output_detail_level`|string|可选|输出详细程度，可取 `brief`、`standard`、`detailed`。|

**最小输入格式**

```JSON
{
  "original_question": "阿尔茨海默病的关键致病机制是什么？"
}
```

## 8\. 模块 4：证据梳理 Agent

### 8\.1 目标

将每个候选假设与支持、反对、不确定证据绑定，形成可追踪证据链，并进行深度质量评审。

### 8\.2 输入

读取 `task_context.hypothesis_cards`、`task_context.evidence_cards`、`task_context.literature_cards`。

```JSON
{
  "hypothesis_cards": [],
  "evidence_cards": [],
  "literature_cards": []
}
```

### 8\.3 输出

### 8\.4 处理流程

|步骤|操作|说明|
|---|---|---|
|1|证据绑定|将 hypothesis\\\_cards\\\[\]\.based\\\_on\\\_evidence\\\_ids 与 evidence\\\_cards 匹配|
|2|证据验证|重新验证每条证据的 support\\\_direction，不信任模块2的预分类|
|3|质量评估|4维评分：直接性\(30%\)、可靠性\(25%\)、充分性\(25%\)、适用性\(20%\)|
|4|缺口识别|对比假设 predictions vs 实际证据，找关键缺口|
|5|综合判定|计算 evidence\\\_strength\\\_score，判断 needs\\\_more\\\_evidence|
|6|生成反馈|输出 detailed\\\_review，包含回退建议|

### 8\.5 评审标准

|维度|权重|评分标准|
|---|---|---|
|直接性|30%|证据是否直接验证假设的核心预测|
|可靠性|25%|来源可信度、方法严谨性、样本量|
|充分性|25%|证据量是否足够支撑结论|
|适用性|20%|研究条件与假设目标的匹配度|

## 10\. 总控最终审核 final\_review

模块 5 完成后，总控进行整体审核。

```JSON
{
  "final_review": {
    "passed": true,
    "overall_score": 0.81,
    "strengths": [
      "假设有明确证据来源",
      "研究计划具有可验证指标",
      "能够说明失败后的下一轮迭代方向"
    ],
    "weaknesses": [
      "因果证据仍需增强"
    ],
    "revision_required": false
  }
}
```

总控最终审核重点：

```Plaintext
1. 研究问题是否清楚
2. 假设是否来自证据和知识空白
3. 每个假设是否能追溯到证据
4. 是否区分支持、反对和不确定证据
5. 研究计划是否对应具体假设
6. 研究计划是否有变量、方法、数据和验证标准
7. 是否说明失败后如何进入下一轮迭代
```

## 11\. 模块输入输出总表

|模块|输入|输出|
|---|---|---|
|模块 1：问题理解|`user_input`|`question_card`|
|模块 2：知识整合|`question_card`|`literature_cards`, `evidence_cards`, `knowledge_gaps`|
|模块 3：候选假设生成|`question_card`, `evidence_cards`, `knowledge_gaps`|`hypothesis_cards`|
|模块 4：证据梳理|`hypothesis_cards`, `evidence_cards`, `literature_cards`|`evidence_map`|
|模块 5：研究计划输出|`question_card`, `hypothesis_cards`, `evidence_map`, `user_constraints`|`research_plan`|
|总控最终审核|全部 `task_context`|`final_review`|

## 12\. 总控校验原则

总控不直接评价每个模块的专业内容质量，而是重点检查：

```Plaintext
1. JSON 格式是否正确
2. 必填字段是否存在
3. 模块自评是否通过
4. overall_score 是否达到阈值
5. 当前输出是否满足下一模块输入要求
6. 是否需要人工确认
7. 是否需要重试、回退或终止
```
